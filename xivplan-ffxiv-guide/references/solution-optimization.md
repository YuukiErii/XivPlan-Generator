# Solution Optimization

本文件定义 skill 在生成攻略图前如何比较解法。Skill 不应只是把用户描述机械地画出来，而应先构造可行方案、排除不安全方案，再根据队伍限制推荐最容易执行的一版。

## 默认优化优先级

除非用户给出更强的固定队偏好，否则按以下优先级处理。

| 优先级 | 目标 | 默认处理 |
|---|---|---|
| 硬门槛 | 机制安全 | 有漏踩、人数不匹配、散开重叠、禁交叉连线却交叉、时间顺序冲突的方案直接淘汰。 |
| 高 | 读条职业和治疗少移动 | 默认让 D4、H1/H2、BLM/PCT/SMN/RDM、需要读条的治疗少跑，或只在自然窗口移动。 |
| 高 | 近战可打到 | D1/D2 和坦克尽量保持 Boss 目标圈内，除非机制强制远离。 |
| 高 | 记忆成本低 | 优先固定站位、固定分组、重复模板，少用临场多条件排序。 |
| 中高 | 容错高 | 优先安全区大、路线不交叉、少同时换位、失误更容易看出来的方案。 |
| 中 | 野队/固定队易沟通 | 规则能写进短宏或一句 callout。 |
| 中 | 图面清晰 | 方案能拆成清楚的多 step 图，不依赖大量交叉线和密集标签。 |

## 新副本未知信息下的保守假设

新副本、占位样例或来源置信度不足时，skill 必须把未知信息显式写进候选方案，而不是把草案包装成定稿。

- 未确认点名规则时，默认按“固定职能可执行”的版本先画 v0.1，并标注需要实战验证的触发条件。
- 未确认安全区大小时，默认选择更大安全边距、更少交叉路线、更少临场换位的候选。
- 未确认移动窗口时，将读条中移动视为高成本；只有时间轴写明 `free` / `preposition` / downtime 时才降低移动惩罚。
- 未确认塔或分摊人数时，候选可以继续生成，但安全检查必须写明“人数假设”，攻略包 unknowns 必须保留。
- 未确认 Boss 是否可攻击时，近战 uptime 只能作为偏好，不能压过机制安全与复位稳定性。

## 权重配置差异

Phase 12 的规划器会根据队伍约束生成不同倾向的候选，同一个机制至少比较以下取舍。

| 场景 | 权重倾向 | 适合输出 |
|---|---|---|
| 开荒优先解 | 容错、记忆成本、安全区面积、复位稳定性优先；允许牺牲少量 uptime。 | 默认推荐，尤其适合未知新副本。 |
| 竞速 / 熟练优化解 | 保 uptime、短路线、读条自然窗口优先；要求队伍能接受更高沟通成本。 | 作为候选或备选，不在未知点很多时默认推荐。 |
| 固定队策略 | 可以使用固定搭档、固定优先级和预先约定的异常处理。 | 适合成员稳定且能复盘的队伍。 |
| 野队可交流策略 | 一句 callout、职能分桶、TH/DPS、小队、八方模板优先。 | 适合公开攻略和临时队伍。 |

推荐结果必须说明采用的是哪一种权重倾向，以及被牺牲的项是什么。

## 候选方案流程

复杂机制按以下流程：

1. 将机制拆成类别：散开、分摊、塔、连线、击退、诱导、Debuff、背对/看向、旋转、分身、延迟、跳舞机制等。
2. 列出已知输入和未知点。未知规则不能当作已确认事实。
3. 先用标准职能站位和标点约定构造一个安全基线方案。
4. 如果存在真实取舍，至少再生成一个候选，例如“最少移动”与“最低记忆成本”。
5. 按默认优先级给每个候选做评分或文字评价。
6. 推荐一个方案，并说明推荐理由和牺牲点。
7. 将推荐方案转成多 step XivPlan 图示计划。

非常简单的机制可以只给一个方案，但仍要跑安全检查和图面清晰检查。

## 建议评分形状

评分是推理辅助，不是死板公式。

```text
solution_score =
  safety_gate
  + caster_movement_score      / 20
  + melee_uptime_score         / 20
  + memory_score               / 20
  + tolerance_score            / 15
  + communication_score        / 10
  + diagram_clarity_score      / 10
  + fight_specific_fit_score   / 5
```

`safety_gate` 是通过/不通过。安全不过的方案不能推荐，即使它移动少或图面好看。

可执行格式、字段说明和命令见 `solution-candidate-format.md`。`scripts/score_solution_candidates.py` 会输出 JSON 与 Markdown 比较报告。

一般移动成本同时作为诊断项输出，用于解释“谁跑得更多”；推荐总分使用专门的 `caster_movement_score`，保持上述权重稳定。

## 移动成本估计

移动成本要按职能和时机估计，不能只看距离。

- `D4` 和已知法系是高成本移动对象。
- `H1/H2` 在机制叠治疗、减伤、复活窗口时移动成本更高。
- `D1/D2` 若离开目标圈、丢身位或必须绕远路，移动成本更高。
- 坦克可以带 Boss，但 Boss 移动可能破坏全队输出、面向和安全区假设。
- 预站阶段的移动成本低于读条中或连续判定中的移动。

职业构成未知时，默认假设：

- D1/D2 是近战。
- D3 是机动性较好的远敏。
- D4 是法系，应少移动。
- 治疗在机制伴随高伤害时不宜长距离强制移动。

## 记忆成本估计

以下情况降低记忆成本：

- 玩家在多个 step 中保持同一个时钟位或同一个半场。
- 分配遵循 TH/DPS、小队、搭档、固定八方等常见模板。
- 同一种颜色、数字或标记在整个机制中含义稳定。
- 优先级短且局部，例如“同色内侧，异色外侧”。

以下情况提高记忆成本：

- 玩家需要同时按多个属性排序。
- 职能在相邻 step 之间无明显原因地换目的地。
- 方案要求读取另一个玩家不易观察的信息。
- 图面需要大量例外说明才能讲清楚。

## 机制类型启发式

### 散开

- 八人散开优先使用固定时钟位。
- 尺寸允许时，让近战保持内圈或可打到的位置。
- 随机点名散开、未点名分摊时，要清楚画出两组并说明点名者如何识别自己。

### 分摊

- 必须写清分摊人数和参与者。
- 分摊点优先靠近场中或下一步复位点，除非机制要求靠边。
- 双分摊优先用小队或 TH/DPS，不优先用临场排序。

### 塔

- 先确认塔数量和每塔人数，再谈优化。
- 塔位置稳定时优先职能固定塔。
- 多人塔优先搭档或小队分配。
- 塔和散开叠加时，优先保证踩塔后的离开路线不撞人。

### 连线 / 线

- 除非机制要求交叉，否则避免交叉线。
- 线源和目标点具有空间意义时，优先按最近安全目的地分配。
- 空间分配不稳定或不可见时，才用职能优先级。
- 图和文字都要说明拉线方向、等待判定、何时复位。

### 击退 / 拉入

- 先确认是否允许防击退。
- 若允许，比较防击退和手动预站，推荐失误率更低的一版。
- 若手动处理，必须画出来源、预站点、落点和击退后复位。

### Debuff / 数字 / 颜色

- 保留玩家可见信息：颜色决定半场，数字决定顺序，倒计时决定轮次。
- 用最短优先级解决冲突。
- 若预站能解决，避免要求玩家在最后一刻换位。

### Limit Cut / 大运动会

- 拆成观察、分配、第一次移动、第二次移动、判定、复位。
- 即使玩家图标缩小，也要保留角色标签。
- 优先复用同一种移动模式，少让每一轮都出现全新路线。

## 推荐输出格式

复杂机制使用以下结构：

```markdown
## 解法选择

### 候选 A：最少移动
- 优点：
- 缺点：

### 候选 B：最低记忆成本
- 优点：
- 缺点：

### 推荐方案
- 推荐理由：
- 牺牲点：
- 需要确认的点：
```

推荐方案确定后，再输出图示拆分：

```markdown
1. 初始站位
2. 机制出现 / 观察点
3. 判断规则
4. 移动路径
5. 判定站位
6. 复位
```

## 安全检查表

推荐方案前必须检查：

- 八个职能都已分配，或缺失分配被明确标为未知。
- 散开距离合理。
- 分摊人数匹配。
- 塔人数匹配。
- 连线端点有效。
- 禁止交叉的路线没有交叉。
- Boss 面向和近战 uptime 假设已说明。
- 队内速记版不与完整说明矛盾。
